## 設計原則

- **CUE に一元化**: スキーマとポリシーは CUE で一度だけ定義する。JSON Schema などの成果物はすべて導出。
- **ライブラリファースト**: gRPC サーバーでラップする前に、まず Go ライブラリ（`sanmon-core`）を作る。
- **テストファースト**: ゴールデンテストが開発を駆動する。各フェーズはテストデータの準備から始める。
- **Lean はメタ証明のため**: Lean が証明するのはポリシー体系の性質であり、個々のルールではない。

## フェーズ 1: CUE スキーマ＋ポリシーの統合とゴールデンテスト

**ゴール**: CUE をスキーマとポリシー両方の唯一の定義元として確立し、後続の開発を駆動するゴールデンテストを整備する。

### タスク

- [x] CUE で基本アクションスキーマを定義（`policy/base/action.cue`）
- [x] ドメイン固有のポリシーを定義（ブラウザ・API・データベース・IaC）
- [x] 構造スキーマを CUE に統合（TypeScript / Zod レイヤーを廃止）
- [ ] ゴールデンテストを整備（ドメインごとに有効 / 無効の JSON フィクスチャ）
- [ ] `cue vet` が有効ケースですべてパス、無効ケースですべて失敗することを確認
- [x] `schema/` の TypeScript コードを削除（不要になったため）

### 成果物
CUE による一元的なスキーマ＋ポリシー定義と、`testdata/` のゴールデンテスト。

## MVP（マイルストーン 0）

ブラウザドメインのエンドツーエンドデモ:

1. CUE でブラウザアクションのスキーマと URL ホワイトリストポリシーを定義
2. CUE から JSON Schema を導出し、制約付きデコーディングに使用
3. Go ライブラリが CUE ポリシーに照らしてブラウザアクションを検証
4. gRPC サーバーでライブラリをリモートから利用可能に
5. ゴールデンテストで既知ケースのポリシー正確性を確認
6. Lean でブラウザポリシーの一貫性と完全性を証明

**成功基準**:
- LLM がホワイトリスト外の URL への `navigate` アクションを生成できない
- 万一生成された場合でも、CUE バリデータが拒否し再生成を促す
- Lean がブラウザポリシーの一貫性とギャップの不在を証明
- `just test` と `just lean-build` ですべて検証可能
