## 設計原則

- **CUEが単一の真実の源**：構造とポリシーはCUEで一度定義。JSON Schemaや他の成果物は導出される。
- **ライブラリファースト**：gRPCサーバーでラップする前に、Goライブラリ（`sanmon-core`）を構築。
- **テストファースト**：ゴールデンテストスイートが開発を駆動。すべてのフェーズはテストデータから開始。
- **Leanはメタ証明のため**：Leanはポリシーシステムの特性を証明する。個別のルールではない。

### タスク

- [x] CUEで基本アクションスキーマを定義（`policy/base/action.cue`）
- [x] ドメイン固有のポリシーを定義（ブラウザ、API、データベース、IaC）
- [x] 構造スキーマをCUEに統合（TypeScript/Zodレイヤーを排除）
- [ ] ゴールデンテストスイートを構築（ドメインごとの有効/無効JSONフィクスチャ）

### 成果物

## MVP（マイルストーン0）

ブラウザドメインのエンドツーエンドデモ：

1. CUEがブラウザアクションスキーマ + URLホワイトリストポリシーを定義
2. CUEから導出されたJSON Schema → 制約付きデコーディング
3. GoライブラリがCUEポリシーに対してブラウザアクションを検証
4. gRPCサーバーがライブラリをリモート使用のためにラップ
5. ゴールデンテストが既知ケースのポリシー正確性を証明
6. Leanがブラウザポリシーの一貫性と完全性を証明

**成功基準**：
- LLMはホワイトリストにないURLへの`navigate`アクションを生成できない
- 万が一生成された場合、CUEバリデータが拒否し再プロンプトをトリガー
- Leanがブラウザポリシーセットの一貫性とギャップなしを証明
- すべて`just test`と`just lean-build`で検証
