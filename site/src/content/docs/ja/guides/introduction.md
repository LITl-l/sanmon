---
title: sanmon とは
description: sanmon の概要と背景
---

**sanmon**（三門）は、AI エージェントが生成するアクションを形式的に検証するためのスタックです。確率的なガードレールではなく、数学的に裏付けられた安全性保証を提供します。

## 解決したい課題

ブラウザ操作、API 呼び出し、DB 操作、IaC — AI エージェントが実行するアクションには実害を伴うものがあります。既存のガードレールは確率的な分類器やパターンマッチに依存しており、見逃しがあっても気づけず、安全性の保証がありません。

## sanmon のアプローチ

LLM の内部構造を解析するのは現実的ではありません。代わりに、sanmon は **LLM の出力** を 3 つの門で制約・検証します。定義はすべて CUE に一元化されています。

```
┌─────────────────────────────────────────────────┐
│  第一門: 制約付きデコーディング (構造的保証)        │
│  CUE から導出した JSON Schema を使い、             │
│  生成時点で構造の正しさを 100% 保証。               │
├─────────────────────────────────────────────────┤
│  第二門: CUE バリデータ (意味的検証)                │
│  ビジネスルールや安全性ポリシーを適用。              │
│  違反があれば拒否し、LLM に再生成を促す。           │
├─────────────────────────────────────────────────┤
│  第三門: Lean 証明器 (健全性証明)                   │
│  ポリシー体系の一貫性・完全性・                     │
│  合成時の安全性を証明。CI でオフライン実行。         │
└─────────────────────────────────────────────────┘
```

CUE がスキーマとポリシーの唯一の定義元（Single Source of Truth）です。JSON Schema は CUE から自動導出され、制約付きデコーディングが確率的な世界（LLM）と決定的な世界（形式検証）をつなぎます。

## 技術スタック

| 要素 | 技術 | 採用理由 |
|---|---|---|
| スキーマ＋ポリシー | CUE | 構造と制約を一元管理できる |
| ランタイム | Go | CUE のネイティブサポート、低レイテンシ、ライブラリとして組み込みやすい |
| 証明 | Lean 4 | モダンな定理証明器。活発なエコシステムと AI との親和性 |
| API | gRPC（REST も可） | フレームワーク非依存で統合しやすい |
| CI | GitHub Actions + Nix | 再現性のある自動検証 |

## 類似プロジェクトとの違い

| プロジェクト | アプローチ | sanmon との違い |
|---|---|---|
| Invariant Labs (Snyk) | Python DSL によるルールベース | 形式証明なし。ランタイムチェックのみ |
| AWS Bedrock Automated Reasoning | 事実内容に対する形式論理 | コンテンツの正しさが対象。アクション制約ではない |
| Guardrails AI | バリデータの集合 | 確率的な検出。数学的保証なし |
| AWS Cedar + Lean | 認可ポリシーの検証 | 静的なポリシー向け。AI エージェントのランタイム制約ではない |
| スマートコントラクト検証 | Coq/Lean/Isabelle でコードを検証 | コード自体が対象。AI が生成するアクションではない |

sanmon は、CUE による一元定義 + 制約付きデコーディング + ランタイム検証 + Lean によるメタ証明を組み合わせた唯一のアプローチです。
